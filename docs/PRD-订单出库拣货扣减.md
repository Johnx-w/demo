## 1. 背景与问题

仓库履约中，销售/出库订单进入系统后，需要形成可执行的拣货任务，并在拣货完成后正确扣减库存。若缺少统一规则，容易出现：

- **超卖**：未预留或并发下重复扣减；
- **账实不符**：完成了任务但库存未扣，或扣了一部分失败留下脏数据；
- **缺货无法收口**：库存不足后任务悬空，预留长期占用，运营无法取消或换货发出；
- **无法审计**：关键操作缺少操作人、结果与幂等追踪。

本模块解决「**出库履约链路**」上的导入 → 待拣 → 完成扣减 → 缺货处置 → 可审计，而不是采购/退货的「入库加库存」。

> 说明：历史口头需求曾称「订单入库」，本 PRD **统一纠正为出库拣货扣减**，避免库存方向错误。

---

## 2. 目标与成功标准

### 2.1 产品目标

1. 运营可导入出库订单，系统自动生成**待拣货任务**并**预留库存**。  
2. 拣货员可完成任务；库存充足时**原子扣减**，不足时进入**可处置异常**且不错误扣减。  
3. 主管/运营可对异常任务执行：**到货重试 / 平替强制完成 / 取消任务与订单**。  
4. 预留默认 **24 小时**超时释放（**仅** `INSUFFICIENT_STOCK` 且仍占用预留时），避免异常单长期锁库；`PENDING_PICK` 本期不因超时释预留。  
5. 全链路具备**权限控制**与**操作日志**（含失败与拒绝）。

### 2.2 成功标准（可验收）

| 编号 | 标准 | 度量方式 |
|------|------|----------|
| S1 | 导入成功后存在 `PENDING_PICK` 任务且 `reserved` 已增加 | 用例 / 集成测试 |
| S2 | 完成成功后 `on_hand` 减少、`reserved` 释放，任务/订单为 `COMPLETED` | 同上 |
| S3 | 完成时不足：不扣 `on_hand`，状态为 `INSUFFICIENT_STOCK`，返回缺口明细 | 同上 |
| S4 | 同 `action_id` 重放不产生第二次扣减或第二次建单 | 幂等测试 |
| S5 | 无权限操作被拒绝且写 `DENIED` 日志，业务数据不变 | 权限测试 |
| S6 | 异常可走通重试 / 平替 / 取消之一收口；超时仅对 `INSUFFICIENT_STOCK`+held 释预留 | 闭环演示 |

### 2.3 非目标（本期不做）

见第 4 节 Out of Scope。

## 3. 用户与场景

### 3.1 角色（Persona）

| 角色 | 典型诉求 | 关键权限 |
|------|----------|----------|
| 仓库运营 | 导入订单、查看任务 | `order.import`, `picking.view` |
| 拣货员 | 完成拣货、到货后重试 | `picking.view`, `picking.complete` |
| 仓库主管 | 平替强制完成、取消任务与订单 | `picking.force_complete`, `picking.cancel` 等 |
| 系统（`system`） | 执行预留超时释放 | 内部调度，非业务登录角色 |

### 3.2 核心用户旅程

1. **运营**导入出库订单 → 系统预留库存并生成待拣货任务。  
2. **拣货员**在列表中看到任务 → 标记完成。  
   - 库存足 → 扣减完成；  
   - 库存不足 → 进入异常，展示缺口。  
3. **主管/运营**在异常工作台选择：等到货重试、平替发出、或取消订单。  
4. 若异常任务长时间无人处理 → **系统**在 24h 后释放预留，任务仍可人工决策。

## 4. 范围

### 4.1 本期包含（In Scope）

1. 订单导入 → 生成待拣货任务（可列表/详情查看）  
2. 拣货完成 → 按规则原子扣减库存  
3. 库存不足 → 任务置为 `INSUFFICIENT_STOCK`，整单不扣减  
4. 异常人工决策：到货重试、平替强制完成、取消任务与订单  
5. 预留 24h 超时释放（兜底；**仅**异常态 `INSUFFICIENT_STOCK`+held）  
6. 基础权限与操作日志（并入业务能力，不单独做成「纯技术故事」）

### 4.2 本期不包含（Out of Scope）

- 收货「入库加库存」、上架、盘点、调拨  
- 波次拣货、路径优化、PDA / 硬件对接  
- 波次级或全局替代料策略、正常路径的拆单部分发运（正常完成采用**全有或全无**；**异常场景下的平替强制完成属于本期**）  
- 替换公司现有完整订单中心 / 商品主数据 / 账号中心（Demo 用本地订单、主数据镜像与权限 Stub 模拟集成）

### 4.3 假设与依赖

| 假设 | 说明 |
|------|------|
| 商品、仓库主数据已存在 | 本域只引用 `warehouse_id` / `sku_id` |
| 统一账号与权限中心存在 | 生产对接权限码；Demo 用 Stub |
| 单仓履约上下文 | 订单带明确 `warehouse_id` |
| Demo 目的 | 展示需求分析、建模、一致性与闭环设计能力，贴近真实仓配，而非完整 WMS |

## 5. 功能需求

功能按交付增量划分，详细验收条件见关联故事文档；此处给出产品级需求描述。

### 5.1 FR-01 导入出库订单并生成待拣货任务

- 拥有 `order.import` 的用户可提交出库订单（外部单号、仓库、明细行）。  
- 系统校验可用库存（`raw_available`）；充足则**同事务**创建本地订单、拣货任务、库存预留与超时任务。  
- 任务状态为 `PENDING_PICK`，订单状态为 `PICKING`。  
- 重复导入同一 `external_order_no` + `warehouse_id` 不得重复建单或重复预留。  
- 导入时库存不足：不产生成功任务与预留；**必落** `IMPORT_FAILED` + 失败操作日志（禁止静默全回滚丢审计）。  
- 已存在 `IMPORT_FAILED` 允许同键重入：以本次明细**全量替换**订单行后再校验。

### 5.2 FR-02 查看拣货任务

- 拥有 `picking.view` 的用户可按状态、仓库、外部单号等筛选任务列表，并查看详情与行缺口/平替信息。

### 5.3 FR-03 完成拣货并扣减库存

- 拥有 `picking.complete` 的用户可将 `PENDING_PICK` 任务标记完成。  
- 判定充足时使用 **effective_available**（计入本任务已预留量），避免「已预留却被判不足」。  
- 库存充足：任务与订单变为 `COMPLETED`，扣减 `on_hand` 并释放对应 `reserved`，写库存流水与终态动作标识。  
- 支持乐观并发：版本冲突时提示任务已变更，且本方不扣库存。

### 5.4 FR-04 完成时库存不足进入异常

- 任一明细可用量不足时：**整单不扣减**，任务变为 `INSUFFICIENT_STOCK`，返回各 SKU 缺口。  
- 异常期间默认**继续占用预留**，供后续重试或人工决策。  
- 异常任务在列表中可筛选可见。

### 5.5 FR-05 异常处置与超时兜底

- **到货重试**：库存恢复后可将任务完成扣减；若预留已被超时释放，须先重新预留再扣减（成功则终态，job=`SKIPPED`；仍不足则不改库存）。  
- **平替强制完成**：指定平替 SKU 与数量（本期 `substitute_qty` 必须等于 `demand_qty`）；释放原 SKU 预留（仅 held 时），扣减平替 SKU 的 `on_hand`；任务/订单为 `FORCE_COMPLETED`。  
- **取消**：取消任务与本地订单（入口：`INSUFFICIENT_STOCK`），释放仍占用的预留，并发出取消类集成事件（Outbox）。  
- **超时**：自预留起默认 24 小时；**仅**当任务为 `INSUFFICIENT_STOCK` 且 `reservation_held=true` 时释放预留，**不改变**任务状态；之后仍可重试/平替/取消。`PENDING_PICK` 本期不因超时释预留。

### 5.6 FR-06 权限与操作审计

- 所有写操作校验权限码；拒绝时业务数据不变，并记 `DENIED`。  
- 每次业务动作写入操作流水：操作人、时间、动作类型、对象、仓库、结果、原因码、请求/响应摘要；支持幂等键追踪。

## 6. 业务规则（产品级）

### 6.1 库存与预留

```
raw_available       = on_hand - reserved - locked
own_reserved        = 本任务对该 SKU 仍占用的预留合计
effective_available = raw_available + own_reserved
账面约束：on_hand ≥ reserved + locked
```

| 场景 | 充足条件 |
|------|----------|
| 导入预留 / held=0 再预留 | 每一 SKU（按 sku 聚合后）`raw_available ≥ demand` |
| 完成 / held=1 重试 | 每一 SKU `effective_available ≥ demand` |
| 平替扣减平替 SKU | `raw_available(substitute) ≥ substitute_qty` |

| 规则 | 说明 |
|------|------|
| 预留 | 导入成功时按 SKU **聚合**增加 `reserved`（同行多 SKU 先求和再锁行） |
| 扣减 | 完成/重试成功：`on_hand` 减少并释放 `reserved` |
| 异常占预留 | `INSUFFICIENT_STOCK` 期间默认继续占用 |
| 平替 | 不扣原 SKU 的 `on_hand`，只释原预留（held 时）并扣平替 `on_hand`；本期全量平替 |
| 缺货策略 | **全有或全无**，禁止正常路径部分发运 |
| 超时 | 默认 24h；扫描以超时任务表的到期时间为准；**仅** `INSUFFICIENT_STOCK`+held |

### 6.2 状态（正式枚举）

**拣货任务**

| 状态码 | 中文含义 |
|--------|----------|
| `PENDING_PICK` | 待拣货 |
| `INSUFFICIENT_STOCK` | 库存不足（可决策） |
| `COMPLETED` | 已完成（原品扣减） |
| `FORCE_COMPLETED` | 强制完成（平替发出） |
| `CANCELLED` | 已取消 |

**本地订单**

| 状态码 | 中文含义 |
|--------|----------|
| `IMPORTING` | 导入中 |
| `IMPORT_FAILED` | 导入失败 |
| `PICKING` | 拣货中 |
| `COMPLETED` | 已完成 |
| `FORCE_COMPLETED` | 平替完成 |
| `CANCELLED` | 已取消 |

`COMPLETED` / `FORCE_COMPLETED` / `CANCELLED` 为任务终态，不可再完成、重试、强制完成或取消。

### 6.3 幂等与并发

- 客户端每次写请求携带 `action_id`；相同键**重放返回首次结果**（含失败与拒绝），不得当作新的成功副作用。  
- 导入另有业务幂等键：`external_order_no` + `warehouse_id`。  
- 状态变更使用版本号 CAS；成功进入终态写入 `terminal_action_id`，防止重复扣减。

### 6.4 原子性

状态变更、库存变更、操作日志、库存流水（若有），以及导入时的订单与超时任务，必须在**同一事务**内成功或全部回滚。

**写序（定稿）：** 终态先 CAS `picking_task`（含 `terminal_action_id`）→ 改 `inventory_balance` → **先**插入 `picking_task_action` 取 id → **再**插入 `inventory_ledger(task_action_id)` → 更新订单/超时 job/outbox。

## 7. 非功能需求

| 类别 | 要求 |
|------|------|
| 一致性 | 禁止「任务已完成但未扣库存」或「已扣库存但任务未终态」的提交结果 |
| 并发 | 同一任务并发完成，至多一次成功扣减 |
| 审计 | 日志追加只写；业务侧不可改删；拒绝与失败也要记 |
| 可恢复 | 导入失败必落 `IMPORT_FAILED` + FAILED 日志，避免长期「有单无任务」且可同键重入 |
| 可观测（Demo） | 可通过任务状态、库存流水、操作流水演示账实与审计 |
| 性能 | Demo 不设严苛 SLA；设计上避免全表无索引扫描关键列表（按状态+时间） |
| 安全 | 生产对接账号中心；Demo 以用户标识模拟登录，权限码鉴权 |
